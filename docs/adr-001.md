# ADR-001: On-Premises Deployment and Policy Enforcement Architecture

**Date:** 2026-02-25
**Status:** Proposed
**Owner:** AI-PM
**Stakeholders:** FDE, AI-SE, AI-SEC, AI-DA

---

## Context

EuroHealth Insurance AG requires a unified AI helpdesk platform with:

- **On-premises deployment only** — non-negotiable constraint from CIO Hans Müller (data sovereignty, EU AI Act, no cloud egress of employee data)
- **EU AI Act compliance** before August 2026 board review
- **€180K budget** for a 6-month engagement
- Operations across 8 EU countries (EN / DE / CZ language support)

The RAG-based helpdesk agent processes employee HR queries, some of which may expose PII (salary records, personal IDs) if the knowledge base is not properly scoped. A governance enforcement mechanism is therefore required in the pipeline before any response reaches the end user.

---

## Decision

**We will implement a Policy Enforcement Point (PEP) as a middleware layer between the LLM generation step and response delivery.**

Policy rules are authored in YAML files (`policies/`), loaded at startup by a Policy Decision Point (PDP), and evaluated against every generated response at runtime.

Supporting decisions:

| Decision | Choice | Rationale |
|----------|--------|-----------|
| LLM runtime | On-prem (Llama 3 or Mistral 7B) | No cloud egress; on-prem GPU constraint |
| Vector store | Qdrant (on-prem) | Self-hosted; supports multilingual embeddings |
| Policy engine | Custom PDP/PEP in Python (MVP) | Avoids external OPA dependency in Phase 1; reviewable by auditors |
| Policy format | YAML files in `policies/` directory | Human-readable for legal/compliance review; machine-parseable at runtime |
| Audit log | Append-only JSON lines in `audit.log` | Immutable; reconstructable; exportable for regulatory review |

---

## Consequences

### Positive

- Every AI response is checked against governance rules before delivery
- Policy changes are deployable without code changes (edit YAML, restart)
- Audit trail is board-defensible and supports EU AI Act Article 13 (transparency)
- On-prem architecture satisfies data sovereignty requirement

### Negative / Trade-offs

- Requires GPU hardware procurement (estimated €15–30K) — not yet in €180K budget
- Custom PEP must be maintained; no vendor SLA
- PEP adds latency (~50–100ms per evaluation) — acceptable for helpdesk use case
- Phase 1 PEP is keyword-matching only; semantic PII detection deferred to Phase 2

### Risks

| Risk | Mitigation |
|------|------------|
| GPU capacity insufficient for on-prem LLM | FDE to validate infrastructure in Week 1 — this ADR is contingent on that validation |
| PEP rule gaps allow PII leakage | AI-SEC to define minimum 5 rules covering PII, scope, escalation before go-live |
| Audit logs contain PII | Log only `query_hash`, not raw query; PII fields never logged in plaintext |

---

## Interface Contracts Required

This decision creates the following cross-role interface obligations:

```
FDE → AI-SEC:  PEP calls PDP via internal function (not REST in Phase 1)
               Input:  { query: str, response_draft: str, confidence: float }
               Output: { decision: "allow"|"block"|"redirect"|"escalate",
                         policy: str, rule_id: str, substitute: str|null }

AI-SEC → FDE:  Policy files in policies/*.yaml (loaded at agent startup)
               Format: see policies/pii-protection.yaml for schema

FDE → AI-DA:   Audit log entries appended to audit.log (JSON lines)
               Fields: timestamp, decision, policy, rule_id, reason, query_hash
```
