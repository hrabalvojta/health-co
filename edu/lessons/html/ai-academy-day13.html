🌎 Real-World Context — How This Works on an Actual Engagement

Today's 60-minute sprint compresses what happens over 2–4 weeks on a real project. Here's what that looks like in practice — and what your role in it actually is.

📁 The skeleton is pre-built — not co-created

In reality, the Lead Architect (FDE) or Engagement Lead (AI-PM) prepares the folder structure and template files before the team starts work — exactly like the skeleton you downloaded today. It comes from a firm-standard template, not improvised in a kickoff meeting.

👤 Each role contributes their section asynchronously

Nobody writes the same document simultaneously. AI-SEC works on governance/policies/ in Week 1–2. FDE designs src/agent.py in Week 2–3. AI-DS builds the golden dataset in Week 3–4. Parallel, asynchronous, in their own sections.

✍️ Who grooms the 4–8 scattered inputs into one coherent deliverable?
AI-PM Primary integrator
Owns docs/ — assembles all role sections into one coherent document, unifies terminology, fills gaps, runs review cycles
FDE Technical reviewer
Checks that architectural decisions from all roles are consistent — no conflicts between what FDE, AI-SE, and AI-SEC designed independently
AI-SEC Governance reviewer
Reviews governance/ section before finalisation — not the integrator, but the approver of anything touching compliance

In practice this plays out over 2–4 review cycles spanning weeks — not a single session. Each cycle: draft → feedback → revise → merge.

💡 So what does today's 60 minutes actually train?

Not the mechanics of file management — the thinking: What does my role produce for others? What do I need before I can design my part? Where does policy enforcement sit in my component? That alignment thinking is what makes weeks of async work coherent. Without it, 8 people produce 8 disconnected documents that nobody can integrate. Today's sprint is the kickoff alignment — the shared mental model the whole team works from.

🏭 Kyndryl Agentic Framework (KAF) — How Today Maps to the Standard

Every design decision you make today corresponds to a named layer in the Kyndryl Agentic Framework. Use this table as your alignment filter — if your component doesn't fit a KAF layer, it likely isn't architected correctly.

KAF LayerWhat it isWhere it lives in Day 13
Agent Core The core reasoning and execution unit of the AI agent FDE RAG pipeline — src/agent.py + src/retriever.py
Orchestration Controls agent flow, sequencing, and multi-step coordination Pipeline flow: Query → Retrieval → Generation → PEP → Response. Sprint checkpoints validate the orchestration design.
Governance Rules, policies, and constraints that bound agent behaviour AI-SEC governance/policies/*.yaml — enforced at runtime by PDP/PEP
Trust & Audit Evidence trail proving every decision was made within policy AI-SE src/audit_logger.py + governance/evidence/audit-log-schema.json
Human Loop Structured mechanisms for human review, override, and approval Escalation rules in governance/policies/escalation-rules.yaml + governance/operating-model/human-override-protocol.md
Agent Registry Enterprise-wide inventory of all agents: owner, risk level, declared policies governance/compliance/ai-inventory.md — the EuroHealth helpdesk agent and the 3 shadow AI systems are registered here. KAF requires every agent to be discoverable, owned, and risk-classified.

Note: This agent is one node in a broader KAF-orchestrated workflow. In Phase 2, it will coordinate with a ServiceNow integration agent and a knowledge base refresh agent under the same Orchestration layer. Design for extensibility now.

📜 Policy as Code — YAML Examples

If your policies exist as a PDF on someone's desktop, they are not policies — they are suggestions. Real governance requires rules that a machine can read, evaluate, and enforce in milliseconds.

policies/pii-protection.yaml 
policy:
  name: pii-protection
  version: "1.0"
  description: Block responses containing employee PII
  rules:
    - id: block-salary-data
      if: response.contains(salary, personal_id, home_address)
      then: block
      log: "PII attempt blocked"
      escalate_to: human_agent
    - id: block-contact-info
      if: response.contains(personal_phone, personal_email)
      then: redact
      log: "PII redacted"
policies/scope-limitation.yaml 
policy:
  name: scope-limitation
  version: "1.0"
  rules:
    - id: block-non-it-topics
      if: query.topic NOT IN [password_reset, vpn, software_install, ...]
      then: redirect
      respond: "I can only help with IT helpdesk topics."

Policy Rules at a Glance

IF response contains salary, personal_id, home_address
THEN block response + escalate to human agent
BLOCK
IF query topic NOT in [password_reset, vpn, ...]
THEN redirect user with scoped message
REDIRECT
IF confidence score < threshold
THEN escalate to human agent
ESCALATE
🔒 Enforcement Flow — PDP/PEP Explained with a Real Scenario

PDP (Policy Decision Point) = the brain. It reads policy rules and decides: allow, block, or redirect. PEP (Policy Enforcement Point) = the muscle. It sits in the pipeline and enforces the PDP's decision before any response reaches the user.

Scenario: The Salary Leak That Didn't Happen

Query: Employee from Prague: "What is the salary of my manager, John Smith? I need it for the budget review."

Step 1 — Retrieval: The RAG agent retrieves Confluence pages, including one with manager salary data from an HR document that was accidentally indexed.

Step 2 — Generation: The LLM generates: "John Smith's monthly salary is 10,000 EUR..."

Step 3 — PDP Decision: The Policy Decision Point evaluates against pii-protection.yaml. Rule block-salary-data fires. Decision: BLOCK.

Step 4 — PEP Enforcement: The original response is suppressed. Safe substitute returned: "I cannot share personal salary information. Please contact HR directly." Event logged. Human agent notified.

Result: PII never exposed. Audit log proves compliance. All in under 200ms.

💡
PDP vs PEP — remember the split. Separating decision from enforcement means you can update rules without touching enforcement code, and test enforcement without changing rules.
3 Most Common Anti-Patterns
  • No PEP in the pipeline. Query → Retrieve → Generate → Response. No enforcement point = any response goes through unfiltered.
  • Policies disconnected from code. AI-SEC wrote YAML. FDE built RAG. Neither talks to the other. Governance on paper only.
  • No audit trail. Policies enforced, but nothing logged. When the auditor asks "what happened March 15 at 14:32?" — silence.
