Code is the last pillar, not the first. In consulting, you never write a line of code until docs/ tells you WHAT to build and governance/ tells you HOW safely to build it. Junior engineers start with code. Senior consultants start with documentation and governance — the code follows naturally.

The Sequence That Matters

Once documentation and governance artifacts are in place, technical implementation follows a clear path:

1 Read your architecture spec — docs/architecture/ defines what you build docs/
2 Check governance constraints — governance/policies/ defines what you CANNOT do governance/
3 Implement within guardrails — src/ is the code that follows both src/
4 Validate against contracts — tests/ proves compliance and correctness tests/

Beyond Compliance: Agent Behavior Governance
EU AI Act coverage is necessary but not sufficient. KAF expects you to govern agent behavior, not just models. Include language like:
  • Policy-as-code constraints — guardrails encoded directly into agent runtime
  • Escalation thresholds — human-in-the-loop triggers defined per agent type
  • Full agent decision logs — every action audit-ready, replayable, attributable

This connects AI-Native Security, Data Sovereignty, and the LLM Orchestrator into one consistent governance model.

❌ Weak Pitch
"We think this AI solution will be really helpful. The technology is exciting and we believe it could save time. We haven't finalized the numbers yet, but the team is confident."
✅ Strong Pitch
"Our L1 auto-resolve pipeline will handle 40% of IT tickets, saving 2.1 FTE (EUR 168K/year). Latency: 1.8s on-prem. FRIA complete. We request Go for Phase 1: 300 users, EUR 120K, 10 weeks."
No-Go is a normal outcome. A No-Go with clear conditions is infinitely more useful than a reluctant Go that hides problems. The best consultants welcome a tough committee.

Go vs No-Go Criteria

GO

  • Clear, realistic design
  • Technology choices justified
  • Governance addressed
  • Cross-role dependencies identified
  • Risks named with mitigations

NO-GO

  • Ignores constraints (cloud for on-prem)
  • No governance consideration
  • No connection to other roles
  • Risks not addressed
  • "It will be fine" without evidence

The Decision Spectrum

Go Proceed as planned
Go Phase 1 Reduced scope first
Conditions Fix issues, then Go
Pause Needs more research
No-Go Fix & resubmit
💡 "Go with Conditions" is the most common real-world outcome. Never present just Go and No-Go — always offer the full spectrum.

Agentic Control Model — Our solution uses a layered agent model that answers Hans's implicit question: “What type of autonomy am I actually approving?”

This layered model reduces perceived risk, demonstrates KAF maturity, and shows exactly how autonomy is controlled at each level.
